# /author-ticket "<free-text idea>"

## Purpose
Entry point for **developers / EMs / tech-leads** to turn an idea into a **complete, `/implement`-ready JIRA ticket tree** — one parent (`Story|Task|Bug`) plus the sub-tasks they list — using the AI-brains to route each node to the right building block, repos, project, and board. The mirror-image of `/implement`: it *produces* the leaves the developer later *consumes*. A plain single ticket is just a tree with no sub-tasks.

## Inputs
- A free-text feature/bug idea, **plus optionally a list of sub-tasks** (inline — one per line, optional `repo:` prefix — or entered interactively). The AI **enriches** each listed sub-task (routes it, drafts problem + AC); it **never invents** sub-tasks. `--suggest` asks the brain to *propose* a breakdown (one sub-task per affected repo) for you to accept/edit at Gate-P.
- Type & epic: `--type <Story|Task|Bug>` (else the AI classifies the parent and you confirm at Gate-P); `--bug` = shorthand for `--type Bug`; `--epic <KEY>` / `--epic "name"` links an **existing** epic (else a Gate-P shortlist, or no link); `--create-epic` creates a new epic (the only way one is created). `--project <KEY>` overrides the resolved project. `--board <name>` pre-selects a board. `--reconfigure` re-runs first-run setup.
- JIRA access via the **Atlassian MCP** if connected in the IDE (preferred — nothing to set); else product-user-local `JIRA_API_TOKEN` / `JIRA_EMAIL` / `JIRA_BASE_URL` (see [auth-and-secrets.md](../jira-integration/auth-and-secrets.md)).
- The **team-local `jira-projects.json`** at the workspace root (product → projects/boards/component map), generated by first-run setup from the plugin-bundled `jira-projects.template.json`. If missing/incomplete → first-run setup. (The real config is never bundled in the plugin — each team's is local, so plugins don't expose other products.)

## Pipeline
**Setup (first run only):** fetch THIS team's product's JIRA projects/boards/components (via Atlassian MCP or REST) and persist the LOCAL `jira-projects.json` at the workspace root (shape from the bundled `jira-projects.template.json`).

**In existing-parent mode** (argument is a Story key / `--parent <KEY>`): step 1 instead **reads + verifies the existing issue** (read-only) and captures its project/type/component; steps 3–4 draft only the sub-tasks (no parent draft/type-classification); step 7 `jira-writer-agent` creates **only the sub-tasks** under `<KEY>` and never touches the parent. Steps 2 (route each sub-task), 5 (completeness), and Gate-P are unchanged.

## Two modes: new parent, or sub-tasks under an existing Story
`/author-ticket` accepts **either** a free-text idea **or** an existing JIRA issue key:

- **New-parent mode (default)** — argument is free text (e.g. `/author-ticket "let shippers see FRLC charges"`). The AI classifies + creates a **new** parent (`Story|Task|Bug`) and the user-listed sub-tasks under it.
- **Existing-parent mode** — argument is an existing **Story/Task/Bug key** (e.g. `/author-ticket EFP-1234`), or `--parent EFP-1234` alongside a free-text note. Use this when a lead already created the Story and a developer wants to **break it into sub-tasks**. Here the command:
  1. **Reads the existing issue read-only** (like `jira-agent`) — verifies it EXISTS and is a valid **sub-task container** (a `Story/Task/Bug`, NOT a `Sub-task` and NOT an `Epic`). If it's a Sub-task or Epic, it refuses with a clear message (sub-tasks can't nest under those). Captures the parent's project, type, summary, and component for context.
  2. **Does NOT draft or create a parent** — the parent already exists. It routes + enriches only the **user-listed sub-tasks** (scoped to the parent's project), runs completeness on each, and Gate-P reviews *the sub-tasks to add under `<KEY>`*.
  3. **`jira-writer-agent` creates ONLY the sub-tasks** with `parent=<EXISTING-KEY>`. It **never edits, transitions, reassigns, or otherwise modifies the existing parent** (`no-destructive-operations` + `jira-write-permissions`) — it only adds children.
  4. Handoff: *"Added N sub-tasks under `<KEY>`. Developer: run /implement on a leaf — `<sub-keys>`."*

  The task-history record is keyed by the existing `<KEY>` (appends a `## Authoring` section for the added sub-tasks). Everything else — routing, enrichment, Gate-P, no-invented-sub-tasks — is identical to new-parent mode.

## Hierarchical authoring (parent + sub-tasks in one flow)
- **User-driven by default.** You list the sub-tasks; the AI routes + drafts each so it is `/implement`-ready. The AI NEVER fabricates a sub-task — unknowns inside one become `TBD — <question>` for you to fill at Gate-P (`no-invented-facts`).
- **The parent is a container; the sub-tasks are the leaves.** `/implement` runs on a leaf (a sub-task, or a childless parent), never on a parent that has sub-tasks — `jira-agent` redirects you to the children (see [`ticket-completeness`](../rules/ticket-completeness.md)).
- **`suggest` mode** (opt-in) proposes a breakdown for you to accept/edit; even then nothing is created without your Gate-P `approve`.
- Sub-tasks are always issuetype **Sub-task**, always `parent`-linked; the parent carries the AI-classified type.

## Fields — createmeta-driven (step 3)
Field existence, required-ness, and allowed values are read **live from JIRA createmeta** for the resolved `project + issuetype` — never hardcoded (see [`implement-intake-format`](../rules/implement-intake-format.md)). The AI fills what it can ground (summary, description, component, issue type, parent, labels); you set the rest at Gate-P (priority, sprint, fix version, story points, team custom fields). **Assignee defaults to YOU** — the authenticated JIRA user running the command — shown at Gate-P and changeable there. **Required-field guard:** fb-tech JIRA requires the custom **Issue Source** (defaults to `User Generated`) — required fields are filled from their default or prompted, so create never fails.

## Task type — AI proposes, you confirm (step 3-4)
The AI classifies the parent from the idea (`fix/broken/reproduce → Bug`; `as-a-user/add/support → Story`; `refactor/migrate/spike → Task`) and you confirm/override at Gate-P or via `--type` / `--bug`. Allowed types come live from createmeta. Sub-tasks are always `Sub-task`. Type drives completeness (Bug ⇒ reproduction steps).

## Epic — link existing (default), never auto-pick (step 4)
Hierarchy: **Epic (L1) → Story/Task/Bug (L0, the parent you create) → Sub-task (L-1)**. Pass the epic via `--epic <KEY>`, `--epic "name"` (AI resolves via JIRA search), an inline mention, OR pick from a **Gate-P shortlist** the AI builds live (`project=<X> AND issuetype=Epic AND statusCategory!=Done`, ranked by component/keyword match). Linking is **optional** (no pick → no link). The AI never auto-picks an epic. `--create-epic` creates a new epic for a larger initiative (Epic + parent + sub-tasks); it is the only way an epic is created. The link mechanism (`parent` field vs "Epic Link" custom field) is discovered via createmeta.

## Product scoping (step 2, before routing)
EFP and RMS share vocabulary (EFP is RMS's rewrite), so generic matching mis-routes (`rate`, `quote`, `charge` exist in both). Routing is therefore **scoped to one product first**, then `router-agent` matches only within that product's brain + `shared-ai-brain` — never across products.
- Resolve the product in order: **`--product` override** → the product user's **remembered default** (per-user local setting) → **ask once** (`EFP / RMS / ATLAS`) and remember.
- This is what stops an EFP idea landing on an RMS building block. ATLAS scopes to `atlas-ai-brain`.

## Project & board resolution (step 3)
1. **Project** = resolved product → `jira-projects.json[product].projects`. One candidate → defaulted; multiple (e.g. EFP's two) → building-block match picks the default, **pre-selected and overridable**.
2. **Board** = **asked** — fetch boards for the chosen project from JIRA, list them (brain may pre-highlight one); product user picks. To make the ticket land on that board, the create step sets the field the board filters on (component/label/sprint) from `component_map`.
3. **Component** = building block → JIRA component (`component_map`).
4. **No building-block match** → service/entity routing → else **Triage**: router shows a best-guess repo shortlist; product user picks; component left blank. Never silently dropped (see `building-block-router`, brain `gaps.md`).

## Gate-P (the one human gate) — reviews the whole tree
Per [`human-approval-gates`](../rules/human-approval-gates.md). The user sees the **entire tree**: parent (type — changeable, title, problem, AC), each sub-task (routing + problem + AC), the resolved project (changeable), board (picked), component per node, the epic to link (from `--epic`/inline/shortlist, or none), and any createmeta-**required** fields still needing a value. At Gate-P the user may **add / remove / split / merge / edit** sub-tasks, confirm or override the type and epic, and set assignee/priority/sprint/fix-version/story-points + team custom fields, then replies `approve | revise <notes> | reject`. The reply is read **by intent, not keyword** (`human-approval-gates`): a clear approval proceeds to create; **any substantive feedback or edit — even without the word "revise"** (e.g. *"drop the third sub-task and route the second to fb-iqs"*) → treated as a revise that re-runs `ticket-composer-agent` with that message; an ambiguous reply → one clarifying question first (never create on unclear input). Every override (re-route, type change, project switch, sub-task edit) is captured verbatim in the record's `## Corrections` log for `/brain-refresh`.

## Outputs
- `<product>-ai-brain/task-history/<PARENT-KEY>.md` with a `## Authoring` section (idea, per-node routing, tree draft, Gate-P decision, project/board/component, epic, sub-task keys) — the SAME file `/implement` later appends to. Pre-key the record lives at `task-history/_drafts/<slug>.md` and is renamed to `<PARENT-KEY>.md` on create.
- On `approve` + create:
  - **new-parent mode:** one new JIRA **tree** — a parent (`Story|Task|Bug`) + one `Sub-task` per approved sub-task (`parent=<PARENT-KEY>`), optionally linked to an existing epic (or a new epic under `--create-epic`), all with the resolved fields.
  - **existing-parent mode:** only the approved `Sub-task`s, each with `parent=<EXISTING-KEY>`. The existing parent is **not modified** in any way.
  - On copy-paste mode: the formatted text, no JIRA write.

## Quality gates
- The drafted tree MUST pass `ticket-completeness` (step 5) before Gate-P — same rule `/implement` refuses without. **Every user-listed sub-task must be leaf-ready** (problem + ≥1 AC, + repro if it's a bug); the parent needs problem + ≥1 AC. The composer never fabricates acceptance criteria and never invents sub-tasks; gaps are surfaced as `TBD` for the user to fill (`no-invented-facts`).
- Creating the tree is a JIRA **write** → opt-in, only after Gate-P `approve`, governed by [`jira-write-permissions`](../rules/jira-write-permissions.md): creating the parent + user-listed sub-tasks and **linking an existing epic** are allowed under `/author-ticket`; creating an epic needs `--create-epic`; deletes/transitions/reassign are forbidden. Creation is **parent-first, then sub-tasks**, and idempotent on stored keys (a re-run never duplicates a created node).
- **Required fields never block create**: `jira-writer-agent` reads createmeta and satisfies every required field (from its default, e.g. **Issue Source** → `User Generated`, or a Gate-P value) before POSTing.
- Project is auto-resolved but always overridable; board is always asked — the command never picks a board silently; the epic is never auto-picked.
- **Component is best-effort**: `jira-writer-agent` verifies each building block's component against the project's *live* JIRA components and sets it only if present; a removed/renamed component is omitted and **never blocks issue creation**.
- Banners per `agent-attribution` at every step.

## Resumability
Re-invoking `/author-ticket` reads `last-phase` from the draft/`<PARENT-KEY>.md` frontmatter and resumes from the next phase (`idea-intake → brain-routing → project-resolved → tree-composed → completeness-checked → gate-P → jira-created`). On a partial create failure, resume re-enters `jira-writer-agent`, which creates only the not-yet-created nodes (parent/sub-tasks) from their stored keys. See [resumability.md](../jira-integration/resumability.md).
